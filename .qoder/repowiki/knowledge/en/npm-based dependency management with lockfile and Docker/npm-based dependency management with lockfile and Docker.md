This React SPA uses the standard npm ecosystem for dependency management. The project is a small single-page application, so its dependency surface is minimal and straightforward.

**System used**: npm (Node Package Manager) with `package.json` as the manifest and `package-lock.json` as the deterministic lockfile (lockfileVersion 3). No vendoring strategy — dependencies are installed into `node_modules` at build time.

**Key files**:
- `package.json` — declares runtime deps (`react`, `react-dom`, `react-icons`) and dev deps (`@vitejs/plugin-react`, `oxlint`, `vite`, TypeScript type packages). All versions use caret ranges (`^x.y.z`), allowing minor/patch updates within major version.
- `package-lock.json` — generated by npm v7+ (lockfileVersion 3); contains exact resolved versions and integrity hashes for every transitive dependency, ensuring reproducible installs across environments.
- `Dockerfile` — copies `package*.json` first, then runs `npm install` to cache the dependency layer separately from source code, improving rebuild speed in CI/container builds.

**Architecture & conventions**:
- Single `package.json` at repo root; no workspaces or monorepo structure.
- Dependencies split cleanly between `dependencies` (runtime: React + icons) and `devDependencies` (build tooling: Vite, Oxlint, types).
- No private registry configuration, `.npmrc`, or `GOFLAGS`/`GOPRIVATE` equivalents — all packages come from the public npm registry.
- No `vendor/` directory or vendored third-party code; everything is pulled from npm on install.
- Node engine requirement is implicit via the `node:20-alpine` base image in the Dockerfile rather than an `engines` field in `package.json`.

**Rules developers should follow**:
- Add new packages under the correct section (`dependencies` vs `devDependencies`).
- Commit both `package.json` and `package-lock.json` together — never regenerate the lockfile without committing it.
- Use caret (`^`) version ranges as already done; avoid pinning to exact versions unless there is a security or compatibility reason.
- When updating, run `npm install` locally to refresh the lockfile before committing changes.
